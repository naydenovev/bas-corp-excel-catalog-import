## Context

Платформа BAF 8.3.23.2299, конфигурация BAS Бухгалтерія КОРП 2.1.5.4,
управляемые формы. Известный отраслевой аналог — типовая внешняя обработка
1С «Загрузка данных из табличного документа» (распространяется отдельно через
ИТС/портал 1С для разных конфигураций, не входит в состав BAS Бухгалтерія
КОРП «из коробки»): выбор объекта (справочник/документ), сопоставление колонок
табличного документа с реквизитами, поиск существующих элементов, импорт.
Наша обработка реализует тот же класс задачи как самостоятельная внешняя
обработка (без заимствования кода типовой, т.к. её исходники не поставляются
с конфигурацией и не являются частью `bascorpxml`), адаптированную под
зафиксированные на этапе explore требования (двойной способ чтения Excel,
явный диалог подтверждения расхождений, протокол изменений).

Рассмотренные варианты архитектуры:

| Вариант | Описание | Оценка |
|---|---|---|
| Минимальные изменения | Жёстко закодированный список из 2-3 справочников с фиксированной картой реквизитов | Быстро, но не решает заявленную задачу — нужен любой справочник |
| Чистая архитектура | Универсальный движок с провайдерами чтения, отдельным слоем валидации, шиной событий | Избыточно для одной обработки без переиспользования в других подсистемах |
| **Прагматичный баланс (выбран)** | Универсальное построение карты реквизитов через `Метаданные`, два простых читателя Excel за общим интерфейсом, императивный модуль формы без лишних абстракций | Покрывает требование «любой справочник» без архитектуры под гипотетическое будущее |

## Goals / Non-Goals

**Goals:**
- Загрузка элементов ЛЮБОГО справочника конфигурации (включая иерархические и
  с табличными частями) из Excel без программирования под конкретный
  справочник.
- Безопасная повторная загрузка одного файла: дубликаты не создаются, найденные
  элементы обновляются только по явному подтверждению пользователя.
- Полная прослеживаемость загрузки — протокол показывает, что было создано,
  что обновлено и что пропущено.
- Работа без обязательной зависимости от MS Excel (режим без COM — основной).

**Non-Goals:**
- Автосоздание отсутствующих групп/связанных элементов по ссылочным
  реквизитам — несовпадение кода считается ошибкой строки (см. proposal).
- Загрузка документов, регистров — только справочники.
- Удаление/архивация элементов, отсутствующих в файле.
- Многопоточная/фоновая загрузка больших файлов (длительные операции БСП) —
  вне рамок первой версии; обработка рассчитана на разовые загрузки
  (десятки–тысячи строк синхронно).

## Decisions

### 1. Построение карты реквизитов через рефлексию метаданных
При выборе справочника на форме модуль обработки строит список реквизитов из
`Метаданные.Справочники.<Имя>.Реквизиты`, плюс системные `Код`, `Наименование`,
`Родитель` (если `Иерархический = Истина`), плюс реквизиты каждой табличной
части (`Метаданные.Справочники.<Имя>.ТабличныеЧасти.*.Реквизиты`) с префиксом
имени ТЧ. Тип реквизита (`ОписаниеТипов`) определяет, доступен ли для него
способ сопоставления «по коду» (для `СправочникСсылка.*`) или прямое значение
(строка/число/дата/булево/перечисление).

**Альтернатива**: чтение через жёстко заданные списки реквизитов на справочник
— отклонено, не масштабируется на «любой справочник».

### 2. Два независимых читателя Excel за общим контрактом
Модуль формы вызывает функцию `ПрочитатьИсходныеДанные(Файл, СпособЧтения)`,
возвращающую единую структуру — массив строк, где строка — соответствие
«Имя колонки → Значение» (через `ТабличныйДокумент` ячейки A1.. читаются как
шапка для имён колонок). Два варианта реализации под капотом:
- `СпособЧтения.ТабличныйДокумент`: `Новый ТабличныйДокумент; .Прочитать(Файл)`
  — платформенный разбор xlsx/ods/xls без COM.
- `СпособЧтения.COMExcel`: `Новый COMОбъект("Excel.Application")`, обход
  `Cells(Row, Col)`, явное `Quit()` и `ОсвободитьОбъект()` в блоке
  `Попытка...Исключение...Иначе` (защита от зависших процессов EXCEL.EXE при
  ошибке середины чтения).

Оба пути отдают одинаковую структуру дальше по конвейеру — диалог
сопоставления и логика записи не зависят от способа чтения.

**Альтернатива**: только COM — отклонено (требует Excel на клиенте, не
работает в тонком клиенте); только ТабличныйДокумент — отклонено (не
покрывает явное требование пользователя на выбор способа, на случай сложных
файлов с XLS-специфичными особенностями).

### 3. Поиск дублей — пользовательский ключ, точное совпадение
Пользователь отмечает чекбоксом одну или несколько строк таблицы
сопоставления как «ключ поиска». Перед обработкой файла для отмеченных
реквизитов строится запрос (`Справочники.<Имя>.НайтиПоРеквизиту` либо
`СтруктураПоиска` через `Справочники.<Имя>.СоздатьЭлемент()` +
`ОбщегоНазначения.ПолучитьЗначенияСвойствОбъектов`/прямой запрос языка
запросов с условиями равенства по ключевым полям). Сравнение — точное
(без учёта регистра для строк, через `СтрСравнить`/`ВРег`), без fuzzy-matching.

**Альтернатива**: автоматическое определение ключа (Код/Наименование) —
отклонено по решению пользователя на этапе explore: гибкость важнее, разные
справочники имеют разные естественные ключи.

### 4. Диалог расхождений и протокол
Для найденного элемента сравниваются значения по ВСЕМ сопоставленным
реквизитам (не только ключевым). Если хотя бы одно значение отличается —
формируется список расхождений `(Реквизит, ЗначениеВБазе, ЗначениеВExcel)` и
показывается модальным диалогом (`ПоказатьОповещение`/отдельная форма списка
с флажками «Применить»). После подтверждения — изменения применяются и
строка протокола добавляется в табличный документ-протокол на форме
(`Элемент | Действие (Создан/Обновлён/Без изменений/Ошибка) | Детали`).
Протокол не пишется на диск автоматически — доступен для просмотра/печати/
сохранения пользователем через стандартные команды табличного документа.

### 5. Подключение через БСП без изменения конфигурации
Регистрация через `СведенияОВнешнейОбработке()` в модуле объекта (паттерн БСП
«Дополнительные отчёты и обработки»), вид — `ЗаполнениеДанных` или отдельная
команда открытия формы без объекта-получателя (загрузка не привязана к
конкретному объекту-владельцу, а к виду справочника, выбираемому на форме).

## Risks / Trade-offs

- **[Риск] Большие файлы (десятки тысяч строк) при синхронном выполнении на
  клиенте/сервере → может превысить таймауты тонкого клиента.**
  → Митигация: чтение и подготовка данных на сервере (`&НаСервере`), запись
  пакетами с промежуточным освобождением памяти; в README отметить
  рекомендуемый верхний предел строк для синхронного режима.
- **[Риск] COM Excel — зависший процесс EXCEL.EXE при необработанном
  исключении.**
  → Митигация: обязательный `Попытка...Исключение` с `Quit()` и
  `ОсвободитьОбъект()` в каждой ветке, включая `Исключение`.
- **[Риск] Несовпадение формата кода ссылочного реквизита (числовой код vs
  строковый, ведущие нули) → ложные «не найдено».**
  → Митигация: сравнение кода после приведения к строке и `СокрЛП`; явное
  сообщение об ошибке строки с указанием искомого кода.
- **[Риск] Реквизиты ТЧ через разделитель в одной ячейке — конфликт, если сам
  разделитель встречается в данных (например, ФИО через запятую при
  разделителе «,»).**
  → Митигация: фиксированный разделитель `;` (точка с запятой), задокументировать
  в README/справке к обработке; в будущей версии — настраиваемый разделитель.
- **[Trade-off] Без длительных операций БСП — обработка проще, но блокирует
  интерфейс на время загрузки.** Принято осознанно для первой версии (Non-Goal).

## Migration Plan

Не применимо — новая внешняя обработка, не затрагивает существующие данные
или код конфигурации. Откат — прекращение использования обработки (удаление
из списка дополнительных обработок), без побочных эффектов на конфигурацию.
Сама загрузка данных — это запись обычных элементов справочников; откат
конкретной загрузки выполняется штатными средствами 1С (ручная правка/удаление
загруженных элементов), протокол загрузки облегчает определение, что именно
было создано/изменено.
