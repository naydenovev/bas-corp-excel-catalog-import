## ADDED Requirements

### Requirement: Выбор вида справочника
Обработка SHALL предоставлять на форме выбор вида справочника из числа всех
справочников конфигурации BAS Бухгалтерія КОРП, доступных текущему
пользователю по правам.

#### Scenario: Пользователь выбирает справочник
- **WHEN** пользователь открывает обработку и выбирает значение в поле «Вид
  справочника» из списка `Метаданные.Справочники`
- **THEN** форма строит и отображает таблицу сопоставления реквизитов этого
  справочника с колонками файла

### Requirement: Построение списка реквизитов
После выбора справочника обработка SHALL построить полный список доступных
для сопоставления реквизитов: простые реквизиты, реквизиты ссылочного типа,
реквизит `Родитель` (если справочник иерархический), реквизиты каждой
табличной части (с явным указанием, что реквизит относится к табличной
части).

#### Scenario: Иерархический справочник
- **WHEN** выбранный справочник имеет признак `Иерархический = Истина`
- **THEN** в списке реквизитов появляется строка «Родитель» с возможностью
  сопоставить её с колонкой Excel

#### Scenario: Справочник с табличными частями
- **WHEN** у выбранного справочника есть одна или несколько табличных частей
- **THEN** реквизиты каждой табличной части показываются в списке отдельной
  группой с префиксом имени табличной части

### Requirement: Выбор способа чтения исходного файла
Обработка SHALL позволять пользователю выбрать на форме один из двух способов
чтения исходного файла: через `ТабличныйДокумент.Прочитать()` (без COM) или
через `COMОбъект("Excel.Application")` (требует установленного MS Excel и
толстый клиент).

#### Scenario: Чтение без COM
- **WHEN** пользователь выбрал способ «Без Excel» и указал путь к файлу
  xlsx/ods/xls
- **THEN** обработка читает файл через `ТабличныйДокумент` и строит из него
  массив строк с колонками, без обращения к OLE/COM

#### Scenario: Чтение через COM Excel
- **WHEN** пользователь выбрал способ «Через MS Excel» и указал путь к файлу
- **THEN** обработка открывает файл через `COMОбъект("Excel.Application")`,
  построчно считывает ячейки и в любом случае (включая исключение) завершает
  процесс Excel (`Quit`) и освобождает COM-объект

#### Scenario: COM Excel недоступен
- **WHEN** пользователь выбрал способ «Через MS Excel», но клиентское
  приложение не толстый клиент либо Excel не установлен
- **THEN** обработка показывает понятное сообщение об ошибке и не пытается
  выполнить загрузку

### Requirement: Сопоставление колонок реквизитам
Обработка SHALL позволять сопоставить каждый реквизит из построенного списка
с одной колонкой прочитанного файла (по заголовку или по букве/номеру
колонки), включая реквизиты табличных частей и `Родитель`.

#### Scenario: Сопоставление простого реквизита
- **WHEN** пользователь указывает колонку Excel для текстового/числового/
  датового реквизита
- **THEN** при загрузке значение этой колонки построчно записывается в
  соответствующий реквизит элемента

#### Scenario: Сопоставление ссылочного реквизита по коду
- **WHEN** пользователь сопоставляет колонку с реквизитом ссылочного типа
  (`СправочникСсылка.*`) или с реквизитом `Родитель`
- **THEN** обработка трактует значение ячейки как код элемента/группы
  связанного справочника и ищет его через `НайтиПоКоду`

#### Scenario: Код связанного элемента не найден
- **WHEN** значение ячейки для ссылочного реквизита/`Родитель` не находится по
  коду ни одним элементом связанного справочника
- **THEN** строка считается ошибочной, элемент по этой строке не создаётся и
  не обновляется, ошибка фиксируется в протоколе с указанием строки файла и
  искомого кода

#### Scenario: Сопоставление реквизита табличной части
- **WHEN** пользователь сопоставляет колонку с реквизитом табличной части
- **THEN** значение ячейки трактуется как список значений, разделённых
  символом `;`, и каждое значение становится отдельной строкой табличной
  части элемента в порядке следования в списке

### Requirement: Выбор ключа поиска дублей
Обработка SHALL позволять пользователю отметить один или несколько
сопоставленных реквизитов как «ключ поиска» для проверки наличия элемента в
базе.

#### Scenario: Ключ не отмечен
- **WHEN** пользователь не отметил ни одного реквизита как ключ поиска
- **THEN** обработка не позволяет запустить загрузку и показывает сообщение о
  необходимости указать хотя бы один ключ

#### Scenario: Поиск по одному ключу
- **WHEN** отмечен один реквизит как ключ (например, «Код»)
- **THEN** для каждой строки файла обработка ищет элемент справочника с точным
  совпадением значения этого реквизита

#### Scenario: Поиск по нескольким ключам
- **WHEN** отмечено несколько реквизитов как ключ (например, «Код
  контрагента» + «Код договора»)
- **THEN** обработка ищет элемент с точным совпадением ВСЕХ отмеченных
  реквизитов одновременно

### Requirement: Создание нового элемента
Если по ключу поиска элемент не найден, обработка SHALL создать новый элемент
справочника и заполнить все сопоставленные реквизиты значениями из строки
файла.

#### Scenario: Элемент не найден — создание
- **WHEN** поиск по ключевым реквизитам не находит существующего элемента
- **THEN** обработка создаёт новый элемент справочника, заполняет
  сопоставленные реквизиты (включая табличные части и `Родитель`) и
  записывает его, фиксируя строку протокола «Создан»

### Requirement: Обнаружение и подтверждение расхождений
Если по ключу поиска элемент найден, обработка SHALL сравнить значения всех
сопоставленных реквизитов в базе со значениями из файла и, при наличии
расхождений, запросить у пользователя подтверждение перед применением
изменений.

#### Scenario: Найден без расхождений
- **WHEN** элемент найден по ключу и все сопоставленные значения совпадают со
  значениями в базе
- **THEN** обработка не изменяет элемент и фиксирует строку протокола «Без
  изменений»

#### Scenario: Найден с расхождениями — подтверждение
- **WHEN** элемент найден по ключу и хотя бы одно сопоставленное значение
  отличается от значения в базе
- **THEN** обработка показывает диалог с перечнем расхождений (реквизит,
  значение в базе, значение в файле) и ждёт от пользователя подтверждения,
  какие из них применить

#### Scenario: Расхождения подтверждены частично или полностью
- **WHEN** пользователь подтвердил применение одного или нескольких
  расхождений в диалоге
- **THEN** обработка обновляет только подтверждённые реквизиты найденного
  элемента, записывает его и фиксирует строку протокола «Обновлён» с
  перечнем фактически изменённых реквизитов

#### Scenario: Расхождения отклонены
- **WHEN** пользователь отказался применять расхождения для конкретного
  элемента
- **THEN** элемент не изменяется, в протоколе фиксируется строка «Пропущен
  (расхождения отклонены)»

### Requirement: Протокол загрузки
Обработка SHALL вести протокол загрузки, содержащий для каждой обработанной
строки файла итоговое действие (Создан / Обновлён / Без изменений / Пропущен /
Ошибка) и детализацию (для обновлений — список изменённых реквизитов, для
ошибок — описание причины).

#### Scenario: Просмотр протокола после загрузки
- **WHEN** загрузка файла завершена (успешно или с ошибками по части строк)
- **THEN** обработка показывает на форме табличный документ-протокол со
  строкой на каждую обработанную строку файла
